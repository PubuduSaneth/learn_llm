# Mastering Context Engineering with Google ADK

This tutorial explores how to move beyond "spaghetti context" by using the Google Agent Development Kit (ADK) patterns found in your Python scripts.

## Objectives

By the end of this tutorial, you will be able to:

- Decouple static system instructions from dynamic per-turn directives to optimize token usage.
- Implement Context Caching to reduce latency and costs for production-scale agents.
- Build a "Steering" Controller that compiles a clean working context for the model.
- Integrate Intent-Based Routing to provide targeted goals and corrective feedback at runtime.

## Concept: Static vs. Dynamic Instruction Separation

The Problem: Traditional LLM apps send the entire system prompt (rules, schemas, safety) with every single turn. This is expensive and slow.
The ADK Solution: Separate Static Instructions (invariant policies) from Turn Instructions (immediate goals).

From your code (app_setup.py):

We define a STATIC_POLICY_HEADER that never changes. This contains the "Laws of the Land."

# Static: The "who" and "how" that stays the same
STATIC_POLICY_HEADER = """You are a strict policy assistant...
Follow this exact JSON schema...
Safety: Never provide medical or legal advice..."""

agent = Agent(
    name="policy_agent",
    static_instruction=STATIC_POLICY_HEADER, # Invariant
    instruction="Default: be concise..."      # Initial dynamic turn
)


2. Concept: Operationalizing Context Caching

The Problem: Even if you separate instructions, the LLM provider still has to "re-read" the system prompt every time, leading to high latency.
The ADK Solution: Use ContextCacheConfig to tell the infrastructure to keep the static part "warm" in the model's memory.

From your code (app_setup.py):

The App object manages the lifecycle of the cache.

app = App(
    name="policy_qa_app",
    context_cache_config=ContextCacheConfig(
        ttl_seconds=3600,     # Keep the static header cached for 1 hour
        min_tokens=1000       # Only cache if the prompt is large enough to matter
    ),
    root_agent=agent
)


Engineering Impact: By caching the 1000+ token policy header, subsequent turns only bill you for the tiny user_message and turn_instruction.

3. Concept: The "Compiled View" (Steering)

The Problem: Shoveling raw history into the window causes "Lost in the Middle" syndrome.
The ADK Solution: Treat the prompt as a rendered view. Use a "Controller" (the SteeringInputs class) to compile a clean set of directives for the current turn only.

From your code (steering.py):

Instead of an unstructured string, we use a structured dataclass to represent the "Knobs" we want to turn.

@dataclass
class SteeringInputs:
    goal: str             # Current mission
    style: str            # Aesthetic/Vibe
    max_cites: int        # Constraint enforcement
    tenant_hint: str      # Context filtering (e.g., EU vs US)
    corrective: str       # Feedback loop for self-correction


The build_turn_instruction function acts as the Compiler, turning these clean variables into a optimized instruction string for the LLM.

4. Concept: Intent-Based Routing & Error Correction

The Problem: Agents often ignore instructions if the user message is complex.
The ADK Solution: Analyze the intent before the agent runs, and inject specific corrective feedback if the last turn failed.

From your code (chat_handler.py):

The chat function performs "Just-in-Time" context assembly.

def chat(session_id: str, user_message: str):
    # 1. Intent Mapping: Clean the goal before the agent sees it
    intent = route_intent(user_message)
    goal = INTENT_TO_GOAL.get(intent)

    # 2. Corrective Steering: If the last turn broke a rule, tell the agent EXPLICITLY
    corrective = get_last_validation_error(session_id) 
    
    # 3. Compiling the View
    turn_instruction = build_turn_instruction(
        SteeringInputs(goal=goal, corrective=corrective, ...)
    )

    # 4. Execution: Update the dynamic window only
    agent.instruction = turn_instruction
    return agent.run(user_message=user_message)


Summary of the ADK Workflow

Define the Foundation: Put rules and schemas in static_instruction (app_setup.py).

Activate the Cache: Set a TTL so you don't pay for the foundation twice (app_setup.py).

Create the Knobs: Use a data structure to define how the agent should behave this turn (steering.py).

The Assembler: Write a handler that maps user intent to those knobs, ensuring the LLM only sees the specific context it needs to win this turn (chat_handler.py).
